# Архитектура проекта

Каркас проекта основан на структуре `gta-reversed`, но адаптирован под ваши требования.

## Ключевые требования

- Графический API: `DirectX 11`.
- Базовое окно приложения: `1920x1080`.
- Цель по производительности: использовать весь доступный потенциал CPU.
- Модель выполнения: многопоточная, с распределением задач по всем ядрам процессора.
- Код и документация: вести на русском языке.

## Корневые директории

- `source` - исходный код приложения на C++.
- `Vendor` - сторонние зависимости.
- `docs.md` - архитектурные решения и правила разработки.

## Обозреватель решений Visual Studio (папки в проекте)

Имя проекта в решении — **Client**, это нормально: так задано в `premake5.lua` (`project "Client"`). Физические каталоги (`source/app`, `source/core` и т.д.) **не обязаны** совпадать с именем проекта.

Папки внутри узла **Client** в обозревателе берутся не с диска напрямую, а из фильтров MSBuild:

- файл `Build/Client.vcxproj.filters` задаёт виртуальное дерево (например `source\app`, `source\core`);
- в `premake5.lua` для проекта **Client** задан блок `vpaths { ... }`, чтобы при следующем запуске `premake` те же фильтры генерировались автоматически.

Если после правок все файлы снова «схлопнулись» в один список — либо отсутствует или устарел `Client.vcxproj.filters`, либо нужно заново сгенерировать проекты через `win-create-projects.bat` / `premake5 vs2022`.

## Порядок узлов в решении (почему Vendor мог быть «первым»)

В обозревателе решений Visual Studio **папки решения** (solution folders) нередко отображаются **выше обычных проектов**, которые не лежат ни в какой папке. Раньше `Client` был без группы, а `discord-rpc` / `imgui` — внутри папки `Vendor`, из‑за этого узел **Vendor** мог визуально оказываться **выше** `Client`, хотя в `.sln` сначала шёл `Client`.

Сейчас в `premake5.lua` проект **Client** обёрнут в группу **`App`**. На верхнем уровне остаются две папки — **`App`** и **`Vendor`**; по алфавиту `App` идёт первой, **`Vendor` — последней**, как вы и хотели.

## Структура исходников

- `source/main.cpp` - точка входа приложения.
- `source/app` - жизненный цикл приложения и верхнеуровневая оркестрация.
  - Имена модулей по мотивам [gta-reversed `source/app`](https://github.com/gta-reversed/gta-reversed/tree/master/source/app) и `source/game_sa` (документация: `docs/CodingGuidelines.md`, обзор классов в `docs/ReversedClasses.md`):
    - `app_scene` — аналог идеи **CScene** (`source/game_sa/Scene.h` в gta-reversed).
    - `app_camera` — как **`app_camera.cpp`** в gta-reversed.
    - `app_light` — как **`app_light`** в gta-reversed.
    - `app_shaders` — слой HLSL/DX11 (в оригинале — пайплайн RenderWare/D3D9, не выделен отдельным `app_*` файлом).
    - `app_input` — как **`app_input`** (клавиатура/пад; в игре также **CPad**, **ControllerState**).
- `source/core` - базовые сервисы (логирование, утилиты, инфраструктура потоков).
- `source/game_sa/GameDataFormat.h` — единый заголовок форматов данных карты (IDE, IPL, gta.dat, IMG, DFF, timecyc, ZON): `e*`, `t*`, `d11::data`, по аналогии с типами в **gta-reversed** (`FileLoader.h`, `tBinaryIplFile` и т.д.).
- `source/game_sa/FileLoader.cpp` — реализации загрузчиков/парсеров в одной единице трансляции (ориентир по имени — `CFileLoader.cpp` в gta-reversed). Включённые каталоги: `source/game_sa` для `#include "GameDataFormat.h"`.
- `source/platform` - платформенный слой (WinAPI, интеграция с `DirectX 11`).
